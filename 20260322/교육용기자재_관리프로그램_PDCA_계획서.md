# 교육용 기자재 관리 프로그램 PDCA 계획서

**프로젝트명:** 교육용 기자재 통합 관리 시스템
**작성일:** 2026-03-22
**버전:** 1.0

---

## 개요

본 계획서는 교육기관의 기자재(노트북, 프로젝터, 실험장비 등)를 효율적으로 관리하기 위한 프로그램 개발에 PDCA(Plan-Do-Check-Act) 사이클을 적용한 문서입니다.

---

## PLAN (계획)

### 1. 현황 분석 및 문제 정의

| 문제점 | 내용 |
|--------|------|
| 기자재 현황 파악 어려움 | 수기 대장 관리로 인한 오류 및 분실 |
| 대출/반납 비효율 | 담당자 부재 시 처리 불가 |
| 유지보수 이력 부재 | 수리 이력 및 비용 추적 불가 |
| 중복 구매 발생 | 기자재 현황 미파악으로 불필요한 구매 |

### 2. 목표 설정

- **정량적 목표**
  - 기자재 분실률 50% 감소
  - 대출/반납 처리 시간 80% 단축 (평균 10분 → 2분)
  - 기자재 활용률 30% 향상
  - 유지보수 비용 20% 절감

- **정성적 목표**
  - 기자재 실시간 현황 파악 가능
  - 담당자 없이도 셀프 대출/반납 가능
  - 데이터 기반 구매 의사결정 지원

### 3. 시스템 요구사항 정의

#### 3.1 핵심 기능 (Must Have)
- 기자재 등록/수정/삭제 (CRUD)
- 대출/반납 관리
- 재고 현황 실시간 조회
- 사용자 권한 관리 (관리자/일반 사용자)
- 기자재 검색 및 필터링

#### 3.2 부가 기능 (Nice to Have)
- QR코드/바코드 스캔 지원
- 유지보수 이력 관리
- 통계 및 리포트 생성
- 이메일/SMS 알림 (반납 기한 초과 등)
- 모바일 웹 지원

#### 3.3 비기능 요구사항
- 동시 접속자 100명 이상 지원
- 응답 시간 3초 이내
- 데이터 일일 백업
- 브라우저 호환성 (Chrome, Firefox, Edge)

### 4. 기술 스택 선정

| 구분 | 기술 | 선정 이유 |
|------|------|-----------|
| Frontend | React + TypeScript | 컴포넌트 재사용, 타입 안전성 |
| Backend | Python (FastAPI) | 빠른 개발, 자동 API 문서화 |
| Database | PostgreSQL | 관계형 데이터 관리, 안정성 |
| 인증 | JWT + OAuth2 | 표준화된 인증 방식 |
| 배포 | Docker + Nginx | 환경 일관성, 확장성 |

### 5. 데이터 모델 설계

```
[사용자] 1 --- N [대출기록]
[기자재] 1 --- N [대출기록]
[기자재] 1 --- N [유지보수이력]
[기자재] N --- 1 [카테고리]
[기자재] N --- 1 [위치]
```

**주요 테이블:**
- `users` - 사용자 정보 (ID, 이름, 학번/사번, 권한)
- `equipment` - 기자재 정보 (관리번호, 품명, 규격, 상태, 위치)
- `categories` - 기자재 분류 (노트북, 프로젝터, 카메라 등)
- `loans` - 대출/반납 기록 (대출자, 기자재, 대출일, 반납예정일, 반납일)
- `maintenance` - 유지보수 이력 (수리내용, 비용, 업체, 날짜)

### 6. 개발 일정 (WBS)

| 단계 | 기간 | 주요 작업 |
|------|------|-----------|
| 1단계: 환경 구축 | 1주차 | 개발환경 설정, DB 스키마 설계 |
| 2단계: 기자재 관리 | 2~3주차 | 기자재 CRUD, 카테고리 관리 |
| 3단계: 대출/반납 | 4~5주차 | 대출 신청, 반납 처리, 상태 관리 |
| 4단계: 사용자 관리 | 6주차 | 로그인, 권한, 프로필 |
| 5단계: 통계/리포트 | 7주차 | 대시보드, 보고서 출력 |
| 6단계: 테스트/배포 | 8주차 | QA, 버그 수정, 운영 배포 |

## DO (실행)

### 1. 개발 실행 전략

#### Sprint 1 (1~2주차): 기반 구축
- [ ] 개발 환경 세팅 (로컬, Git 저장소)
- [ ] DB 스키마 생성 및 초기 데이터 투입
- [ ] 사용자 인증 API 구현 (로그인/로그아웃)
- [ ] 기본 UI 레이아웃 구성

#### Sprint 2 (3~4주차): 기자재 관리
- [ ] 기자재 등록/조회/수정/삭제 API
- [ ] 기자재 목록 화면 (검색, 필터, 페이지네이션)
- [ ] 기자재 상세 화면
- [ ] QR코드 생성 기능

#### Sprint 3 (5~6주차): 대출/반납
- [ ] 대출 신청 API 및 화면
- [ ] 반납 처리 API 및 화면
- [ ] 대출 현황 조회
- [ ] 연체 알림 기능

#### Sprint 4 (7~8주차): 마무리
- [ ] 통계 대시보드 구현
- [ ] 유지보수 이력 관리
- [ ] 통합 테스트
- [ ] 배포 및 사용자 교육

### 2. 코딩 표준 및 협업 규칙

- Git Flow 브랜치 전략 사용 (main / develop / feature/*)
- 커밋 메시지 컨벤션: `feat:`, `fix:`, `docs:`, `refactor:`
- PR 리뷰 필수 (최소 1명 승인)
- 함수/변수 네이밍: snake_case (Python), camelCase (JS)

### 3. 리스크 대응 계획

| 리스크 | 가능성 | 영향도 | 대응방안 |
|--------|--------|--------|----------|
| 일정 지연 | 중 | 높음 | MVP 기능 우선, 부가기능 후순위 |
| 요구사항 변경 | 높음 | 중 | 주 1회 담당자 미팅 |
| 보안 취약점 | 중 | 높음 | 코드 리뷰, OWASP 기준 적용 |
| 서버 장애 | 낮음 | 높음 | 일일 백업, 장애 복구 절차 수립 |

## CHECK (점검)

### 1. 품질 검증 기준

#### 기능 테스트 체크리스트
- [ ] 기자재 등록 시 관리번호 자동 생성 확인
- [ ] 대출 중인 기자재 중복 대출 방지 확인
- [ ] 권한별 메뉴 접근 제한 확인
- [ ] 반납 기한 초과 시 알림 발송 확인
- [ ] 검색/필터 결과 정확도 확인

#### 성능 테스트 기준
- [ ] 페이지 로딩 3초 이내
- [ ] 동시 접속 100명 시 응답 정상 여부
- [ ] DB 쿼리 실행 시간 1초 이내

#### 보안 점검 항목
- [ ] SQL Injection 방어 확인
- [ ] XSS 공격 방어 확인
- [ ] 인증 토큰 만료 처리 확인
- [ ] 비밀번호 암호화 저장 확인

### 2. 사용자 수용 테스트 (UAT)

**테스트 대상자:** 담당 교직원 3명, 학생 대표 5명
**테스트 기간:** 배포 후 2주
**평가 기준:**

| 항목 | 평가 방법 | 목표 점수 |
|------|-----------|-----------|
| 사용 편의성 | 설문 (5점 척도) | 4.0점 이상 |
| 기능 완성도 | 시나리오 테스트 | 90% 이상 통과 |
| 오류 발생률 | 로그 분석 | 오류율 1% 미만 |

### 3. KPI 측정 방법

- **대출 처리 시간:** 대출 신청~완료 timestamp 차이 측정
- **활용률:** (총 대출 횟수 / 전체 기자재 수 × 365) × 100
- **분실률:** (분실 건수 / 전체 기자재 수) × 100
- **시스템 가동률:** (정상 운영 시간 / 전체 시간) × 100

### 4. 주기별 점검 계획

| 주기 | 점검 항목 |
|------|-----------|
| 일간 | 오류 로그 확인, 백업 성공 여부 |
| 주간 | 대출/반납 현황, 사용자 피드백 수집 |
| 월간 | KPI 측정, 기자재 실물 점검과 시스템 대조 |
| 분기 | 보안 점검, 성능 최적화 검토 |

## ACT (개선)

### 1. 피드백 기반 개선 프로세스

```
사용자 피드백 수집
       |
       v
이슈 우선순위 분류 (긴급/일반/요청)
       |
       v
개선사항 백로그 등록
       |
       v
Sprint 계획에 반영
       |
       v
개발 → 테스트 → 배포
       |
       v
효과 측정 (다음 PDCA 사이클)
```

### 2. 개선 우선순위 분류 기준

| 등급 | 기준 | 대응 기한 |
|------|------|-----------|
| 긴급 (P1) | 시스템 장애, 데이터 손실 | 즉시 (24시간 내) |
| 높음 (P2) | 핵심 기능 오작동 | 3일 내 |
| 보통 (P3) | 불편사항, UI 개선 | 다음 Sprint |
| 낮음 (P4) | 신규 기능 요청 | 로드맵 검토 |

### 3. 2차 개발 후보 기능 (로드맵)

**단기 (3개월 내):**
- 모바일 앱 또는 반응형 웹 최적화
- 바코드/QR 스캔 대출 기능
- 일괄 등록 (Excel 업로드)

**중기 (6개월 내):**
- 기자재 예약 시스템
- 자산 감가상각 자동 계산
- 외부 구매 시스템 연동

**장기 (1년 내):**
- AI 기반 수요 예측 및 구매 추천
- IoT 센서 연동 (위치 추적)
- 타 기관 연계 공유 대출 시스템

### 4. 다음 PDCA 사이클 진입 기준

1차 PDCA 완료 후 아래 조건 충족 시 2차 사이클 시작:
- KPI 측정 결과 보고서 작성 완료
- 사용자 만족도 조사 완료
- 개선 백로그 10개 이상 수집
- 경영진/담당자 검토 회의 완료
